//Data Security

Ab hum Salesforce ka sabse important topic shuru kar rahe hain: Data Security. Iska simple matlab hai ki kis insaan ko kya dikhna chahiye aur kya nahi.

Salesforce security ko 4 Levels mein baanta gaya hai. Isse aap aise samajh sakte hain jaise ek badi building ki security:

1. Organization Level (Building ka Main Gate)
Ye poori company (Org) ki security hai.

Kaise? Ismein tay hota hai ki kaun login kar sakta hai, kab kar sakta hai (Login Hours), aur kahan se kar sakta hai (IP Address).

Example: Office ke ghante khatam hone ke baad koi login na kar paaye.

2. Object Level (Kamre ka Darwaza)
Ye sabse simple level hai. Ismein aap tay karte hain ki kya koi user us table (Object) ko dekh bhi sakta hai ya nahi.

Kaise? Iske liye hum Permission Sets aur Profiles ka use karte hain.

Example: Ek Interviewer "Job Applications" dekh sake, par use Delete na kar sake.

3. Field Level (Kamre ke andar ki Tijori)
Ho sakta hai aap user ko kamre mein aane dein (Object access), par aap chahte hain ki wo kuch khaas cheezein na dekh sake.

Kaise? Field-Level Security (FLS) se hum sensitive fields ko chupa dete hain.

Example: HR manager "Salary" field dekh sake, par baaki employees ko wo field dikhe hi nahi.

4. Record Level (Ek khaas File)
Aapne user ko Object aur Field ka access de diya, par ab aap chahte hain ki wo sirf apne records dekhe, dusron ke nahi. Isse manage karne ke 4 tarike hain:

Org-Wide Defaults (OWD): Ye base level hai. Isse aap data ko "Private" ya "Public" set karte hain.

Role Hierarchies: Ismein manager apne niche kaam karne walon ka data dekh sakta hai.

Sharing Rules: Ye "Exceptions" hain. Agar koi data private hai, toh rules bana kar kisi khaas group ko access diya ja sakta hai.

Manual Sharing: Jab ek record owner khud kisi dusre user ke saath record share kare.

Auditing (Nazar Rakhna)
Sirf lock lagana kaafi nahi hai, nazar rakhna bhi zaroori hai:

Login History: Kaun kab login hua?

Field History Tracking: Kisne field ki value badli? (Puraana aur naya data dono dikhta hai).

Setup Audit Trail: Salesforce ki settings mein kisne ched-chad ki?

Quiz Answers (Aapki help ke liye):
Which feature controls access at the Object level?

Answer: Profiles or Permission Sets (Option mein jo bhi ho).

Which of these is a method for controlling Record-level access?

Answer: Organization-Wide Defaults (OWD).

Aap ye tay karte hain ki company ke passwords kitne surakshit honge.

Steps:

Setup mein Password Policies search karein.

Yahan aap ye badal sakte hain:

Length: Password kitna lamba ho (e.g., 8 ya 10 characters).

Complexity: Kya usme numbers aur special symbols (!@#) zaroori hain?

Expiration: Password kitne din baad expire hoga (e.g., 90 days).

Lockout: Kitni baar galat password daalne par account lock ho jaye.

3. Network Access (Trusted IP Ranges)
Iska matlab hai ki agar user office ke internet se login kar raha hai, toh use Verification Code (OTP) nahi maanga jayega.

Steps:

Setup mein Network Access par jayein.

Ye Network Access se zyada sakht (strict) hai. Agar aapne yahan IP daal di, toh user sirf aur sirf usi IP se login kar payega. Bahar se login karne ka option hi khatam ho jayega.

Steps:

Setup > Profiles > Kisi bhi Profile ko select karein.

Niche scroll karke Login IP Ranges par jayein aur range set karein.

5. Login Hours (Time ki pabandi)
Aap tay kar sakte hain ki employee sirf duty ke hours mein hi kaam karein.

Steps:

Setup > Profiles > Profile select karein.

Login Hours par click karke Edit karein.

Jaise: Monday to Friday, 9:00 AM to 5:00 PM.

Note: Agar koi 5:01 PM par kaam karne ki koshish karega, toh Salesforce use bahar nikal dega ya login nahi karne dega.

## **1. Core Concepts: Access Control**

Salesforce mein data access teen tarikon se manage hota hai:

* **Profiles (The Foundation):** Har user ki ek hi profile hoti hai. Iska use default settings (Apps, Page Layouts, Login Hours) ke liye karein. **Best Practice:** "Minimum Access - Salesforce" profile clone karke use karein.
* **Permission Sets (Task-Based):** Kisi specific task (e.g., Manage Leads) ke liye permissions dete hain. Ek user ko multiple sets mil sakte hain.
* **Permission Set Groups (Persona-Based):** Multiple Permission Sets ko ek "Persona" (e.g., Recruiter) mein group karte hain taaki management aasaan ho.

 **Field-Level Security (Control Access to Fields)** 
 Field-Level Security (FLS) ka matlab hai ki aap decide karte ho ki user ko object ke andar kaunsa specific field dikhega aur kaunsa nahi.

Object Level: Pura "Candidate" folder dikhega ya nahi.

Field Level: Folder ke andar ka "Mobile Number" ya "Salary" dikhega ya nahi.

Steps to Manage Field Permissions (As per your text)
Step 1: Permission Set banana aur Field Access dena
Agar aap chahte ho ki Interviewers candidate ka record update kar sakein par sensitive info (jaise SSN) na dekh payein:

Setup search: Setup mein jao aur Permission Sets search karke select karo.

New Set: New par click karo. Label mein naam do Update Candidate Records. License mein None rakho aur Save karo.

Find Settings: Search box mein Candidates (object ka naam) likho aur use select karo.

Edit Permissions: Edit button par click karo.

Object Permissions: Read par tick karo (taaki wo candidate record dekh sakein).

Field Permissions: Ab niche fields ki list aayegi:

Jo dikhana aur edit karwana hai (jaise Apex aur C# checkboxes), unke Read aur Edit dono par tick karo.

Jo sirf dikhana hai (jaise Hire By date), uska sirf Read tick karo.

Jo chhupana hai (jaise SSN, Email), unhe uncheck rehne do.

Save: Settings save kar do.

Step 2: Multiple Permission Sets ko ek saath update karna
Jab koi naya field banta hai aur aap use bahut saare permission sets mein ek saath set karna chahte ho:

User Management: Setup mein User Management Settings search karo.

Feature On Karo: Field-Level Security for Permission Sets during Field Creation ko Enable (ON) karo.

Object Manager: Object Manager mein jao aur apna object (e.g., Candidate) select karo.

Field Selection: Fields & Relationships mein jao aur us field ko select karo jise manage karna hai (e.g., Job Category).

Set Security: Set Field-Level Security button par click karo.

Mass Update: Saare permission sets ki list dikhegi. Ek hi screen par sabka Read/Edit access set karo aur Save kar do.

Key Takeaways (Important Points)
Security vs Page Layout: Page layout sirf screen se field hatata hai, par FLS field ko Reports, Search aur List Views se bhi gayab kar deta hai (Real protection).

Read-Only: Aap field ko read-only bana sakte ho taaki user dekh toh sake par change na kare.

Best Practice: Salesforce recommend karta hai ki field permissions hamesha Permission Sets se hi manage karein.

//Record Hierarchy
Concept: Record-Level Security Kya Hai?
Permission sets se humne "Object" aur "Fields" control kiye, par Record-Level Security yeh decide karti hai ki user ko kaunse specific records (rows) dikhenge.

Example: Kya ek Recruiter ko sirf apne banaye hue candidates dikhne chahiye ya company ke saare candidates? Iska faisla Record-Level Security karti hai.

Four Ways to Control Record Access (Order: Low to High)
Salesforce mein access hamesha "lock" se "open" ki taraf jata hai:

OWD (Org-Wide Defaults): Sabse pehle data lock karo (Sabse restricted level).

Role Hierarchies: Managers ko subordinates ka data dikhao.

Sharing Rules: Groups ko access dene ke liye automatic exception banayein.

Manual Sharing: Record owner khud kisi ko access de "Share" button se.

Steps: Org-Wide Defaults (OWD) Set Kaise Karein?
OWD ka maqsad hai sabse restricted user ke liye base-level access set karna.

Step 1: Sharing Settings Open Karein
Setup mein jayein.

Quick Find box mein Sharing Settings search karein aur select karein.

Step 2: Edit Defaults
Organization-Wide Defaults section mein upar diye gaye Edit button par click karein.

Step 3: Object Access Choose Karein
Har object ke liye access level chunein:

Private: Sirf record owner aur uske managers dekh sakte hain. (Example: Candidates, Job Applications).

Public Read Only: Sab dekh sakte hain, par edit sirf owner kar sakta hai. (Example: Positions).

Public Read/Write: Sab dekh aur edit kar sakte hain.

Controlled by Parent: Agar object Master-Detail relationship mein hai (Example: Reviews), toh wo apne parent ka access follow karega.

Step 4: Grant Access Using Hierarchies
Agar aap chahte hain ki manager apne niche wale logo ka record dekh sake, toh Grant Access Using Hierarchies ko checked rehne dein. Custom objects ke liye aap ise uncheck karke band bhi kar sakte hain.

Step 5: Save and Recalculate
Save par click karein.

Salesforce data update karega. Ismein thoda time lag sakta hai, aur kaam pura hone par aapko Email notification milega.

Example Scenario (As per your text):
Position Object: Kyunki standard employees ko saari positions dikhni chahiye par wo edit nahi kar sakte, isliye ise Public Read Only rakha gaya.

Candidate/Job Application: Inmein sensitive info hoti hai, isliye ise Private rakha gaya taaki log ek dusre ka data na dekh sakein.

Review Object: Yeh Candidate ya Job App se juda hota hai, isliye iska access Controlled by Parent hota hai.

Key Takeaway:
Hamesha OWD ko "Private" ya sabse zyada "Restrictive" rakhein, aur phir Role Hierarchy ya Sharing Rules ka use karke access ko dheere-dheere "Open" karein.
